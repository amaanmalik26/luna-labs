Executive Summary

Luna Labs is a SvelteKit-based marketing site and internal admin system. The application combines a polished public-facing experience with a server-rendered admin layer that manages leads and portfolio projects through Supabase. The architecture is straightforward and maintainable, with clear separation between public UI, server routes, and data access.

Public experience

A polished marketing site built with SvelteKit and Svelte 5.

Admin layer

Protected pages for leads, portfolio content, and dashboard operations.

Data layer

Supabase powers persistence, auth, and server-side data access.

System Overview

The app is a single SvelteKit application with two main concerns:

  1. Public website experience
  2. Protected admin dashboard for content and lead management

Core Components

Public application

  • Main site shell and navigation live in the routes directory.
  • The homepage renders through the main page entry.
  • Shared UI lives in the reusable component library.
  • The lead capture form is implemented in the request quote form component.

Server-side application layer

  • Request handling is routed through SvelteKit server files.
  • Global request initialization is handled in the server hooks entry.
  • Admin route protection is enforced in the admin layout guard.

Data layer

  • Supabase clients are created through the server-side client factory.
  • Public portfolio reads use a publishable client.
  • Admin and write operations use a service client with elevated access.

Request Flows

Public lead submission

  1. A visitor submits the quote form.
  2. The form validates locally using a shared Zod schema.
  3. The browser sends the payload to the leads API endpoint.
  4. The server validates again, applies IP-based rate limiting, and writes to Supabase.
  5. A Discord notification is emitted.

Admin access

  1. A user visits the admin area.
  2. The request is intercepted by the server hooks layer.
  3. The server checks Supabase auth and whether the email is in the allowlist table.
  4. Access is allowed only if the user is verified and approved.

Data and Persistence Model

The app relies on Supabase tables for structured data, including:

  • leads
  • projects
  • allowed_admins

The database contract is typed in the shared database definitions, and form validation is enforced with Zod schemas for lead and project data.

Authentication and Authorization

Current approach

  • Authentication is handled through Supabase.
  • Server-side auth is initialized per request in the hooks layer.
  • Admin access is gated by a server-side layout check.

Strengths

  • Auth is centralized instead of duplicated across pages.
  • Admin routing is protected by a shared layout-level guard.
  • Supabase SSR cookies allow server-rendered access without storing session state in the browser.

Architectural Strengths

  • Clear separation of concerns between presentation, server routes, validation, and data access.
  • Server-first data access is appropriate for forms, admin editing, and protected operations.
  • Shared validation prevents client/server rule drift.

Risks and Technical Debt

  • Lead submission rate limiting is process-local and will not scale across multiple instances.
  • Admin access is based on a simple allowlist rather than a richer roles model.
  • Operational visibility is still fairly light for external integrations and failures.
  • Automated test coverage is not yet comprehensive for core workflows.

Recommendations

  • Introduce a more durable rate-limiting layer for lead submissions.
  • Add role-based admin permissions if the admin area grows.
  • Increase observability for critical flows such as lead submission and Supabase errors.
  • Expand automated tests around the lead API and portfolio management actions.
